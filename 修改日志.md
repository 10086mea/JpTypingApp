# C++20 协程与 Python 本地后台的架构演进

## 背景与需求
我们的目标是为 `JpTypingApp` 增加一个 AI 助理功能，利用 Gemini 2.5 Flash 提供实时的日文打字补全和语法纠错。由于 AI 接口的网络请求可能会导致明显的延迟，因此我们**绝不能在主线程（ImGui 渲染循环）中阻塞等待网络响应**。

起初，我们考虑过直接在 C++ 中通过 `CreateProcess` 每次按键都拉起一个 Python `exe` 来处理请求。但在实际评估后，发现 Python 解释器的冷启动开销高达数百毫秒，这对于要求高频、低延迟的打字应用来说是绝对不可接受的。

## 架构选型：本地 HTTP 异步微服务
经过权衡，我们彻底否定了“按需拉起进程”的方案，转而采用了**分离的本地服务端架构**：

1. **后端 (API Service)**：用 Python 编写了一个极其轻量的持久化后台服务。我们仅使用了 Python 内置的 `http.server` 和 `urllib`（零第三方依赖），暴露本地 `127.0.0.1:5000` 端口。这不仅规避了 Python 的冷启动性能损耗，而且后续也可以轻易使用 PyInstaller 打包为独立的免部署 Exe 环境。我们还单独剥离了 `config.json` 用来配置 API Key。

2. **前端与网络层 (C++ Client)**：在 C++ 中发起 HTTP POST 请求。起初尝试使用 CMake 的 `FetchContent` 引入 `libcurl` 和第三方 JSON 库，但很快遭遇了复杂的 CMake 依赖与版本兼容性报错（形如 `Compatibility with CMake < 3.5 has been removed`）。为了追求极致的轻量与可控，我们果断**抛弃了庞大的依赖构建**，转而引入了单头文件神器 `cpp-httplib` (httplib.h) 以及 `nlohmann/json.hpp`，把编译复杂度降到了最低。

## C++20 协程 (Coroutine) 极致融合
在网络请求不阻塞的前提下，如果使用传统的回调函数 (Callback) 或单纯的多线程来更新 UI，代码逻辑将被分割得稀碎。而利用 C++20 的无栈协程 (`<coroutine>`) 以及您亲自封装的 `CoroTask`，我们实现了一套极其优雅的异步等待方案。

我们定制了一个 `AsyncHttpPost` (Awaiter)，在其 `await_suspend` 阶段：
1. 立即挂起当前的 C++ 业务逻辑。
2. 剥离一个底层的 `std::thread` 去执行真正的、可能耗时数秒的 HTTP 请求。
3. 请求完成后，通过 `handle.resume()` 原地唤醒逻辑并继续往下执行 JSON 解析，再通过配合 `std::mutex` 锁，将结果安全地推送给 UI 渲染端。

```cpp
// 只需要一行代码，宛如同步调用一般！
std::string response = co_await AsyncHttpPost{text, ""};
```
这就保证了 ImGui 能够以非常高的帧率依然流畅运行，而复杂的网络请求、数据解析则在后台悄无声息地进行。

## 解决实战 Bug：TCP 管道破裂 (WinError 10053)
在联调时，我们遇到了 Python 端频发崩溃断连，报错：
`ConnectionAbortedError: [WinError 10053] 你的主机中的软件中止了一个已建立的连接。`

**问题根源：**
这是一个极其经典的前后端超时不同步问题。Gemini 的流式大文本返回通常需要数秒，而我们在 C++ 客服端的 `httplib::Client` 起初只设置了 5 秒超时 (Timeout)。这就导致当前端等不及先行切断 TCP 断开连接时，后端 Python 刚分析完想把数据回写，引发了向“已关闭套接字”写入数据的致命系统异常。

**最终解决方案：**
1. 在 C++ 端将 `set_read_timeout` 的容忍时间延长至 30 秒，确保慢速网络和复杂的文法推倒有充足的安全空间。
2. 在 Python 的 HTTP Server 的 `do_POST` 方法内部做了 `ConnectionAbortedError` 的防护拦截，保证即便客户端强行退出了，由于有防护网，后端的服务进程依然能稳定常驻，绝不闪退。

## 解决痛点 Bug 2：极致手速触发的协程越界崩溃 (Use-After-Free)
随着自动补全 UI 的上线，我们发现了在高频连续敲击键盘时，程序会直接遭遇 Win32 未处理异常而闪退。

**崩溃逻辑闭环分析：**
在我们的打字防抖逻辑（500ms）设计中，如果在上一次请求还没返回结果时，用户继续敲击代码导致下一次防抖再次触发，C++ 就会通过代码 `m_activeTask.emplace(...)` 创建新的网络请求覆盖旧的任务。
而在 `CoroTask` 的底层 `~Task()` 实现中，这个 `.emplace()` 被调用的瞬间，旧的协程栈结构会被立刻且毫不留情地摧毁 (`coro.destroy()`)！
这就造成了一个悲剧：刚才被该协程剥离 (`detach()`) 出去的那个等待中的后台网络线程依然在使用。当它迟几秒钟终于等来了 HTTP 返回报文，它尝试在最后一行代码去执行 `h.resume()` 强行唤醒协程时——**实际上那个堆栈指针早已成为了野指针**（已经被覆盖销毁）。这种“复活早已死去的变量”的行为引发了极其致命的内存访问违规。

**终极多线程与协程安全令牌 (Cancellation Token)：**
为了让这种基于 `coroutine` 栈的 Fire-and-Forget (即用即脱离) 异步任务拥有工业级的安全性，我们实施了生命周期强解耦：
1. **智能指针代替栈捕获**：禁止后台的网络线程在 lambda 表达式中捕获 `this` 或者堆栈内的任何值。它只被允许持有一群由 `std::shared_ptr` 打包的数据、安全传值副本 (`std::move`) 的入口参数。这意味着，旧线程彻底摆脱了由于母协程栈被提前摧毁带来的数据读写违规隐患。
2. **安全唤醒临界区**：引入了一套自旋双共享锁 `m_cancelMutex` 和 `m_isCancelled` 信号。在主线程准备创建新协程从而销毁旧协程的前端，主线程会顺着获取之前发送出去的锁，拉起取消指令：把旧信号 `m_isCancelled` 置为 `true`。
3. 当那一条慢吞吞的孤儿线程结束了它的 API 请求爬升后，它在即将踏下 `h.resume()` 断头台的那几微秒前，必须先获取 `m_cancelMutex`。一旦发现这个任务已经被判了“取消(`true`)”死刑，它便会静好地自行放弃唤醒流程释放自己，**绝不去触碰已被 C++ 回收的死域协程。**
## 性能优化：API 轻重分离架构 (Decoupling)
在实际体验中，我们发现**高频打字极易触发长达 10秒 左右的严重预测卡顿和 Token 浪费**。

**性能瓶颈剖析：**
1. **生成负担过重**：原版 Prompt 中强制要求模型必须同时完成“字词补全”和“发掘语法错误、中文推敲解释、输出格式化 JSON”。要知道，大语言模型的单句推理耗时高度正相关于其所需生成的字数 (output tokens)。强迫它在每一次键盘按键停顿后都做深度体检，这就偏离了极速打字工具的初衷。
2. **服务端并发排队**：打字时的 0.5s 防抖机制会飞速制造很多“半截请求”(如刚打了几个假名就触发)。虽然我们在 C++ 层面取消了渲染，但残存的 TCP 连接依旧把这些垃圾请求堆积到了 Google 免费版的 API 网关上。这种行为极易触发后端的 Rate Limiting 隐性限流，使得真正有用的请求被迫排队等候长达 10 秒之久。

**终极解决方案：冷热分层、各司其职**
为了彻底治愈预测拥堵梗阻，我们将前后端业务彻底劈开：
1. **轻量极速预测 (completion 模式)**：维持了 0.5s 防抖的触发绑定。我们在 Python 侧设计了专用轻量极简 Prompt，且直接关掉 JSON 返回，硬性限制最大输出上限为 20 个 Tokens (`maxOutputTokens=20`)。这就把原来漫长的 10 秒等待**硬核压缩到了极速的 1 秒左右**！同时避免了无意义残句的大量 Token 付费损耗。
2. **重量级诊断查杀 (grammar 模式)**：将繁琐的 JSON 分析彻底剥离异步事件。转而在 C++ 的 ImGui 模块中嵌入快捷键侦听器 `Ctrl + Enter`。只有在用户输入了长文，心里没底时**主动砸下快捷键**，后端专属沉重的高 Token 分析引擎才会开始全力运转，并将纠错解析呈递给右侧的专属面板。这就像给系统装上了动静分离的涡轮引擎。

## UI 交互与体验重构：打造现代化 IDE 般的输入体验
为了让这个简易应用拥有不输给 Cursor 或 Github Copilot 的手感，我们在不引入任何重型控件库的情况下，仅凭借 ImGui 原生 API 实现了两项强交互的极客体验：

1. **沉浸式内联补全水印 (Inline Ghost Text)**：
   我们摒弃了普通的底部文本提示。通过拦截 `InputTextMultiline` 的输入包围盒坐标（Bounding Box），并在调用 `CalcTextSize` 算出用户已输入文字的实际高度偏移后，使用底层的 `ImDrawList->AddText()` 组件，直接将半透明的绿色补全预测文字“画”在了输入框的**内部背景层**上。使得高亮提示总是自适应跟在你刚敲完的语句下方，做到了沉浸无感的完美伪装。
2. **纯文本可自由选中化 (Selectable ReadOnly)**：
   ImGui 默认的普通文字控件（`TextWrapped` 等）是不支持鼠标拖拽选区复制的。为了让用户可以自由复制右侧 AI 提供的各种修正长句，我们将原本的文本段落强制替换成了附加 `ImGuiInputTextFlags_ReadOnly` 选项的无边框、无底色 `InputText`，通过伪装成静态文本，直接打通了原生的高亮与右键系统的支持；另外搭配显式的 `[📋 一键复制]` 快捷按钮补全了 UX，大幅提升了语言学习用户的摘抄与改错效率。

## 解决痛点 Bug 3：快捷键嵌套死锁触发系统 abort()
在分流架构接入 `Ctrl + Enter` 全句纠错快捷键时，开发中曾遇到按键按下的瞬间，程序直接挂起并报错 `abort() has been called` 的阻断级崩溃。

**崩溃根源分析：** 
由于快捷键的按键检测在渲染左侧面板 `DrawLeftPanel()` 的作用域内进行，而此时渲染函数为了保护文本脏读，必然正持有 `m_uiMutex` 互斥锁。一旦按键满足后直接粗暴调用 `OnGrammarCheckTriggered()` 函数，该函数起手就会为了读取状态而**再次申请同一把全局锁** `std::lock_guard<std::mutex> lock(m_uiMutex)`。在 C++11 标准库中默认的 `std::mutex` 属于非可重入锁（Non-recursive），所以当同一个后台线程二次对同一把钥匙加锁时，直接引发死锁（resource deadlock would occur），被编译器底层安全性捕捉并果断抛出了中断异常。

**修复方案：** 
极其简单但致命的细节：在持有锁的域内仅通过设置一个临时的 `bool trigger_grammar = true;` 录入触发信号。等当前这帧的所有 UI 控件绘制完毕、**离开大括号域释放 `m_uiMutex` 锁之后**，再以单身轻松的状态拉起切换函数，这彻底把这个潜伏的死锁炸弹给拔除了。

## 解决痛点 Bug 4：日语 IME 预输入状态 (Composition State) 频繁触发自动补全
在原版 ImGui 中，使用日语输入法敲击罗马音但未按回车确认上屏时，底层的输入框缓冲也会高频发生改变。这导致了防抖器经常在用户拼字思考时被意外触发，进而引发无效的网络请求。

**修复方案 (Win32 IMM 重挂钩)**：
1. **防抖时长扩展**：将基础防抖阈值 `m_debounceDelay` 延长至更宽裕的 1000ms。
2. **底层 IME 上下文钩子**：引入 `<imm.h>`，使用 `glfwGetWin32Window` 拿到原生 Windows 窗口句柄 `HWND`。通过 `ImmGetContext` 直接读取底层的 `ImmGetCompositionStringW` 拼字串长度。
3. **精准判定打断**：只要该长度大于 0，说明用户正处于假名/罗马音选词状态，系统即刻**拦截并重置防抖定时器**，彻底杜绝了无效的补全请求。

## 总结
通过摒弃每次拉起子进程的笨重操作，采用常驻 本地 HTTP 微服务配合 C++20 协程底层拦截调度的模式，我们用最少的依赖组合（`httplib.h` + `json.hpp`）获得了一套高内聚、低耦合、UI 零阻塞的现代化客户端 AI 驱动架构。

## 新增特性：原生日语语音识别 (WinRT 整合)
为了进一步提升输入效率与交互维度的丰富度，我们引入了 Windows.Media.SpeechRecognition 实现了原生的 C++ 语音转写。
1. **C++/WinRT 零成本协程挂载**：因为我们之前自己手写的 `CoroTask` 在底层已经完美符合 C++20 协程原语标准，所以在引入 WinRT （其同样使用 C++20 协程 `IAsyncOperation` 标准）后，我们可以直接在自定义业务任务 `Task<void>` 里通过 `co_await recognizer.RecognizeAsync()` 非阻塞式挂起录音操作。整个复杂的 API（初始化 COM 单元、麦克风权限请求、调用云端模型推理）代码甚至没有一行包含传统回调或底层线程拉起。
2. **ImGui 无缝重绘 UI**：放弃了系统级别的蓝色 UWP Default UI，将识别状态（即 `MediaCapture` 权限获取完毕及正在聆听）与 ImGui 原生界面完全绑定，保持统一的大厂极客全屏界面风格。

在完成了基础的语法查错和代码预测打字器后，我们对基础核心架构进行了一层降维打击不仅实现了枯燥的编辑器功能，还将它直接拓展成为了**全能多模态二次元 AI 伴侣**！

**实施细节：**
1. **多端点分离重构：** 在 Python 后端 pi_service/gemini_service.py 引入了最新的 google-genai 包，实现原版 1beta API 库和新库的互用。增加了一个 /chat 请求处理，能够在一次会话里使用 gemini-2.5-flash 输出人格扮演日语并直接联动 gemini-2.5-pro-preview-tts，流式导出保存 .wav 媒体块。
2. **C++ 模式切换状态机：** 在 C++ 界面层重构引入 AppMode，在全局头部用优雅的 ImGui RadioButton 让用户能够在沉浸式打字界面与二次元聊天弹幕视图之间一秒切换，历史记录会使用彩色高亮结构体进行缓存渲染 (m_chatHistory)。
3. **设置项开放与防抖调整：** 撤除了写死的常数，现在可以将 m_systemPrompt 和 m_debounceDelayMs 以 SliderInt 和 InputTextMultiline 的形式让用户随心所欲在右侧修改防抖帧宽与 AI 美少女的人设口癖（比如『前辈...』、『欧尼酱...』）。

**潜在困难与攻坚记录：**
- **困难点1：WinMM 并发抢音**。SND_ASYNC 会在下一句音频到来时直接切断前一条，若用户在高潮发问或连续语音识别输入时，大模型返回极快的话容易切断语音碎片。目前暂定这符合快速跳跃打字时的预期。

esume() 去刷新左下角的诊断 UI 或瀑布流消息池。这是 C++20 协程实战的终极性能大考。
    
## 后续进阶：听写模式优化与实时音频缓存重放

随着功能的完善，我们进一步改进了语音识别与聊天的体验极限：

### 1. 从短句命令到原生流式听写 (Dictation & Streaming)
原先使用的 WinRT SpeechRecognizer 默认应用的是短句场景 (WebSearch 引擎)，只要说话者略微停顿就会被立刻中断。为了支持真正的多行口语连讲，我们实施了以下底层改造：
- **听写约束拦截**：在初始化识别器时，强制注入了 SpeechRecognitionScenario::Dictation (长文听写) 约束。系统现在能够更好地结合长文语境自动在句子之间插入逗号和句号，极大提升了长文听写的准确度。
- **动态防抖拉伸**：在 ImGui 的参数面板内新增了自定义滑块，允许用户将 EndSilenceTimeout 手动拉长到最大 10 秒，从此可以一边思考一边说话，不怕被打断。
- **流式视觉证明 (HypothesisGenerated)**：不再是说话完毕才一次性出字。我们通过订阅 HypothesisGenerated 的异步回调，实时把底层刚听到的临时碎句 (Hypothesis text) 直接上推到了界面的状态栏，实现了犹如视频同声传译字幕般的原生流式听写观感。
- **主动打断安全拦截**：如果在听写的中途不想等沉默超时，可以随时点击变为红色的录制按钮。此时程序会安全抽调出挂靠的 IAsyncOperation 或内部的 SpeechRecognizer，强行执行 StopRecognitionAsync()，瞬间截断并挽救已听到的最后一部分文字然后立刻发出。

### 2. 彻底告别单文件覆盖：全尺寸音频溯源 (Audio Replay)
为了实现长长的聊天流里，每一句二次元 AI 对话都可以随时被点击重复播放的要求，我们将简单的文件下发机制拓展成了完整的微小媒体资源管理器：
- **UUID 分布式散列保存**：把后端的 Python 发声模块从单例的 	emp_reply.wav 覆盖重装模式，彻底改为了基于 uuid.uuid4().hex 配合缓存目录 (udio_cache) 分发存储。每一句由网络传回的声音都会永久储存在本地硬盘防丢。
- **C++ ChatMessage 重播指针**：将 C++ 端内的气泡消息队列 m_chatHistory 新增了 udioPath 物理变量挂钩。当用户上拉历史消息滚轮并点击某句话下面的绿色【🔊 重播】按钮时，原生 PlaySoundA 函数将根据这句话私有的 UUID 文件路径瞬间重新唤醒声卡进行高保真的免阻塞复读。
