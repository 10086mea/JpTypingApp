# C++20 协程与 Python 本地后台的架构演进

## 背景与需求
我们的目标是为 `JpTypingApp` 增加一个 AI 助理功能，利用 Gemini 2.5 Flash 提供实时的日文打字补全和语法纠错。由于 AI 接口的网络请求可能会导致明显的延迟，因此我们**绝不能在主线程（ImGui 渲染循环）中阻塞等待网络响应**。

起初，我们考虑过直接在 C++ 中通过 `CreateProcess` 每次按键都拉起一个 Python `exe` 来处理请求。但在实际评估后，发现 Python 解释器的冷启动开销高达数百毫秒，这对于要求高频、低延迟的打字应用来说是绝对不可接受的。

## 架构选型：本地 HTTP 异步微服务
经过权衡，我们彻底否定了“按需拉起进程”的方案，转而采用了**分离的本地服务端架构**：

1. **后端 (API Service)**：用 Python 编写了一个极其轻量的持久化后台服务。我们仅使用了 Python 内置的 `http.server` 和 `urllib`（零第三方依赖），暴露本地 `127.0.0.1:5000` 端口。这不仅规避了 Python 的冷启动性能损耗，而且后续也可以轻易使用 PyInstaller 打包为独立的免部署 Exe 环境。我们还单独剥离了 `config.json` 用来配置 API Key。

2. **前端与网络层 (C++ Client)**：在 C++ 中发起 HTTP POST 请求。起初尝试使用 CMake 的 `FetchContent` 引入 `libcurl` 和第三方 JSON 库，但很快遭遇了复杂的 CMake 依赖与版本兼容性报错（形如 `Compatibility with CMake < 3.5 has been removed`）。为了追求极致的轻量与可控，我们果断**抛弃了庞大的依赖构建**，转而引入了单头文件神器 `cpp-httplib` (httplib.h) 以及 `nlohmann/json.hpp`，把编译复杂度降到了最低。

## C++20 协程 (Coroutine) 极致融合
在网络请求不阻塞的前提下，如果使用传统的回调函数 (Callback) 或单纯的多线程来更新 UI，代码逻辑将被分割得稀碎。而利用 C++20 的无栈协程 (`<coroutine>`) 以及您亲自封装的 `CoroTask`，我们实现了一套极其优雅的异步等待方案。

我们定制了一个 `AsyncHttpPost` (Awaiter)，在其 `await_suspend` 阶段：
1. 立即挂起当前的 C++ 业务逻辑。
2. 剥离一个底层的 `std::thread` 去执行真正的、可能耗时数秒的 HTTP 请求。
3. 请求完成后，通过 `handle.resume()` 原地唤醒逻辑并继续往下执行 JSON 解析，再通过配合 `std::mutex` 锁，将结果安全地推送给 UI 渲染端。

```cpp
// 只需要一行代码，宛如同步调用一般！
std::string response = co_await AsyncHttpPost{text, ""};
```
这就保证了 ImGui 能够以非常高的帧率依然流畅运行，而复杂的网络请求、数据解析则在后台悄无声息地进行。

## 解决实战 Bug：TCP 管道破裂 (WinError 10053)
在联调时，我们遇到了 Python 端频发崩溃断连，报错：
`ConnectionAbortedError: [WinError 10053] 你的主机中的软件中止了一个已建立的连接。`

**问题根源：**
这是一个极其经典的前后端超时不同步问题。Gemini 的流式大文本返回通常需要数秒，而我们在 C++ 客服端的 `httplib::Client` 起初只设置了 5 秒超时 (Timeout)。这就导致当前端等不及先行切断 TCP 断开连接时，后端 Python 刚分析完想把数据回写，引发了向“已关闭套接字”写入数据的致命系统异常。

**最终解决方案：**
1. 在 C++ 端将 `set_read_timeout` 的容忍时间延长至 30 秒，确保慢速网络和复杂的文法推倒有充足的安全空间。
2. 在 Python 的 HTTP Server 的 `do_POST` 方法内部做了 `ConnectionAbortedError` 的防护拦截，保证即便客户端强行退出了，由于有防护网，后端的服务进程依然能稳定常驻，绝不闪退。

## 解决痛点 Bug 2：极致手速触发的协程越界崩溃 (Use-After-Free)
随着自动补全 UI 的上线，我们发现了在高频连续敲击键盘时，程序会直接遭遇 Win32 未处理异常而闪退。

**崩溃逻辑闭环分析：**
在我们的打字防抖逻辑（500ms）设计中，如果在上一次请求还没返回结果时，用户继续敲击代码导致下一次防抖再次触发，C++ 就会通过代码 `m_activeTask.emplace(...)` 创建新的网络请求覆盖旧的任务。
而在 `CoroTask` 的底层 `~Task()` 实现中，这个 `.emplace()` 被调用的瞬间，旧的协程栈结构会被立刻且毫不留情地摧毁 (`coro.destroy()`)！
这就造成了一个悲剧：刚才被该协程剥离 (`detach()`) 出去的那个等待中的后台网络线程依然在使用。当它迟几秒钟终于等来了 HTTP 返回报文，它尝试在最后一行代码去执行 `h.resume()` 强行唤醒协程时——**实际上那个堆栈指针早已成为了野指针**（已经被覆盖销毁）。这种“复活早已死去的变量”的行为引发了极其致命的内存访问违规。

**终极多线程与协程安全令牌 (Cancellation Token)：**
为了让这种基于 `coroutine` 栈的 Fire-and-Forget (即用即脱离) 异步任务拥有工业级的安全性，我们实施了生命周期强解耦：
1. **智能指针代替栈捕获**：禁止后台的网络线程在 lambda 表达式中捕获 `this` 或者堆栈内的任何值。它只被允许持有一群由 `std::shared_ptr` 打包的数据、安全传值副本 (`std::move`) 的入口参数。这意味着，旧线程彻底摆脱了由于母协程栈被提前摧毁带来的数据读写违规隐患。
2. **安全唤醒临界区**：引入了一套自旋双共享锁 `m_cancelMutex` 和 `m_isCancelled` 信号。在主线程准备创建新协程从而销毁旧协程的前端，主线程会顺着获取之前发送出去的锁，拉起取消指令：把旧信号 `m_isCancelled` 置为 `true`。
3. 当那一条慢吞吞的孤儿线程结束了它的 API 请求爬升后，它在即将踏下 `h.resume()` 断头台的那几微秒前，必须先获取 `m_cancelMutex`。一旦发现这个任务已经被判了“取消(`true`)”死刑，它便会静好地自行放弃唤醒流程释放自己，**绝不去触碰已被 C++ 回收的死域协程。**

## 总结
通过摒弃每次拉起子进程的笨重操作，采用常驻 本地 HTTP 微服务配合 C++20 协程底层拦截调度的模式，我们用最少的依赖组合（`httplib.h` + `json.hpp`）获得了一套高内聚、低耦合、UI 零阻塞的现代化客户端 AI 驱动架构。
